# Phase 1：首页与需求提交流程规格

## 1. 阶段目标

Phase 1 先完成老人端最小可演示闭环：

老人打开首页，选择需求类型，进入统一提交页，通过文字、语音、照片说明需求，选择紧急程度，提交后生成工单。

本阶段不做完整后台、不做支付、不做真实派单、不接 DeepSeek。重点是把“老人如何方便表达需求”做清楚。

## 2. 设计原则

本阶段使用 `huashu-design` 的设计原则：

- 不做通用 AI 风格页面，不堆渐变、emoji 和无意义装饰。
- 所有按钮、卡片、输入区必须服务老人操作。
- 首页不做复杂二级分类，减少老人判断负担。
- 语音是核心能力，但首页语音不抢走全部视觉重心。
- 页面语言像社区服务台，不像商业下单平台。

## 3. 首页方向

首页采用“安靖社区服务台”方向。

页面感觉应该像社区党群服务中心的数字化服务窗口，而不是普通电商或外卖平台。首页需要让老师一眼看出：这是社区可以落实的老人需求收集工具。

### 3.1 首页结构

从上到下：

1. 顶部社区服务台
2. 社区公告
3. 四个核心功能卡片
4. 首页语音入口
5. 底部导航

### 3.2 顶部社区服务台

内容：

- 产品名：邻里帮
- 社区：安靖社区服务台
- 值班人员：今日值班 张主任
- 社区电话：028-12345

交互：

- 社区电话按钮可以做模拟拨号提示。
- 不在首页重复放“人工帮助”功能卡片。
- 人工介入能力放在服务台模块和提交页“需要社区电话联系”开关里。

### 3.3 社区公告

首页保留一个轻量公告区域。

示例：

- 本周三下午 3 点：手机使用教学
- 本周五上午：楼道安全巡查

要求：

- 公告不超过两条。
- 不做滚动复杂组件。
- 字体清晰，视觉权重低于功能卡片。

### 3.4 四个功能卡片

中间功能卡片按以下顺序排列：

1. 生活需求
2. 上门事项
3. 外出协助
4. 社区反馈

每个卡片点击后直接进入统一提交页，不再进入二级分类。

#### 生活需求

说明短句：

代买、拿快递、日常小事

适用：

- 买药
- 买菜
- 买饭
- 日用品
- 拿快递
- 其他生活小事

#### 上门事项

说明短句：

维修、保洁、理发等

适用：

- 水电维修
- 管道疏通
- 家政保洁
- 理发剪发
- 家电维修
- 助浴护理

#### 外出协助

说明短句：

就医、办事、散步等

适用：

- 去医院
- 去药店
- 去银行
- 去社区服务中心
- 散步
- 参加社区活动

#### 社区反馈

说明短句：

楼道、电梯、路面问题

适用：

- 楼道灯坏
- 电梯按钮看不清
- 路面不平
- 公告字太小
- 坡道不好走
- 公共设施损坏

### 3.5 首页语音入口

首页需要保留语音入口，但不要做成整页主角。

位置：

- 放在四个功能卡片下方，底部导航上方。

形式：

- 一个横向大按钮或中等圆形按钮。
- 文案：说出您的需要。
- 点击或按住后开始语音识别。

首版交互：

- 识别出“买药、买菜、快递”等，进入生活需求提交页。
- 识别出“水龙头、维修、疏通、保洁”等，进入上门事项提交页。
- 识别出“医院、银行、散步、外出”等，进入外出协助提交页。
- 识别出“楼道、电梯、路灯、路面、公告”等，进入社区反馈提交页。
- 识别失败时提示：没听清，请再说一遍，或点下面的卡片。

## 4. 统一提交页方向

提交页采用“社区值班协助 + 清晰表单”的混合方向。

不采用分步向导，因为分步会增加老人理解负担。也不采用巨型语音按钮，因为提交页还需要照片、紧急程度、电话联系等信息。

### 4.1 提交页结构

从上到下：

1. 顶部返回与需求类型
2. 简短说明
3. 文字说明框
4. 语音说明
5. 上传照片
6. 紧急程度
7. 是否需要社区电话联系
8. 固定底部提交按钮

### 4.2 顶部区域

内容：

- 返回按钮
- 当前需求类型，例如：生活需求
- 副标题：请把事情说清楚，社区值班人员会看到

要求：

- 不写长篇介绍。
- 不出现“我们提供大按钮、少选择”等说明性文字。

### 4.3 文字说明框

标题：

请说清楚您的需求

提示语：

例如：我想买照片里的降压药，家里已经吃完了。

要求：

- 输入框高度足够大。
- 老人可以只填文字，也可以通过语音自动填入。

### 4.4 语音说明

标题：

语音说明

按钮文案：

按住说话

交互：

- 语音识别结果自动写入文字说明框。
- 同时保留“已保存语音留言”的状态，用于后台复核。
- 识别失败时显示“没听清，请再说一遍”。

要求：

- 按钮明显，但不大到压过所有表单。
- 语音不是装饰，必须能推动填写内容。

### 4.5 上传照片

标题：

上传照片

提示语：

可以拍药盒、维修位置、楼道问题等照片。

要求：

- 所有需求类型都支持照片。
- 支持 1 到 3 张。
- Phase 1 可以先显示文件名和缩略图。

### 4.6 紧急程度

选项：

- 不急
- 今天需要
- 很紧急

规则：

- 选择“很紧急”时，默认打开“需要社区电话联系”。
- “很紧急”工单推荐处理方为社区工作人员。

### 4.7 社区电话联系

形式：

- 开关或大按钮。

文案：

需要社区工作人员电话联系我

规则：

- 打开后，工单状态为“待社区电话确认”。
- 不在老人端承诺具体服务完成时间。

### 4.8 提交按钮

文案：

生成工单

要求：

- 固定在页面底部。
- 点击后校验至少有文字说明、语音说明或图片中的一种。
- 如果三者都没有，提示：请先说一下、写一下，或上传一张照片。

## 5. 提交成功页

提交成功后显示工单凭证。

内容：

- 工单已生成
- 工单编号
- 需求类型
- 当前状态
- 推荐处理方
- 是否有语音
- 是否有照片
- 是否需要电话联系

按钮：

- 回首页
- 查看我的工单

提示文案：

您的需求已记录。社区值班人员会根据情况联系邻里互助者、认证服务者或社区工作人员处理。

## 6. 我的工单页

Phase 1 做简版。

列表字段：

- 工单编号
- 需求类型
- 提交时间
- 当前状态
- 推荐处理方
- 是否有语音
- 是否有图片

状态示例：

- 待社区确认
- 邻里互助等待中
- 已转认证服务者
- 待社区电话确认
- 已完成

## 7. 我的页面

Phase 1 做基础版。

内容：

- 当前老人
- 切换老人
- 字体大小：标准 / 大 / 特大
- 紧急联系人
- 社区电话
- 我的工单入口

要求：

- 老人切换在“我的”页面内，不放在首页顶部。
- 字体调节全局生效。

## 8. Phase 1 非目标

本阶段不做：

- 在线支付
- 真实拨号
- 真实派单
- DeepSeek API
- 完整后台管理
- 邻里圈审核
- 费用结算
- 政策问答

## 9. 演示主线

### 9.1 生活需求

1. 首页点击“生活需求”。
2. 进入提交页。
3. 语音说：我想买照片里的降压药。
4. 上传药盒照片。
5. 选择“今天需要”。
6. 提交生成工单。
7. 成功页显示推荐处理方：社区工作人员先电话确认。

### 9.2 上门事项

1. 首页点击“上门事项”。
2. 输入：厨房水龙头漏水，地上很滑。
3. 上传照片。
4. 选择“今天需要”。
5. 提交生成工单。
6. 成功页显示推荐处理方：社区认证服务者。

### 9.3 社区反馈

1. 首页点击“社区反馈”。
2. 输入：2 号楼 3 层楼道灯坏了，晚上看不清。
3. 上传照片。
4. 选择“很紧急”。
5. 系统自动打开“需要社区工作人员电话联系我”。
6. 提交生成工单。

## 10. 验收标准

- 首页是“安靖社区服务台”方向。
- 首页中间 4 个功能卡片顺序为：生活需求、上门事项、外出协助、社区反馈。
- 首页不再重复放“人工帮助”和“电话联系”两个入口。
- 首页有克制的语音入口。
- 点击任意功能卡片可进入统一提交页。
- 提交页包含文字、语音、照片、紧急程度、电话联系开关。
- 提交页不采用复杂分步流程。
- 提交后可生成工单成功页。
- 工单能在“我的工单”看到。
- 字号、按钮、卡片满足适老化。
- 页面不出现说明性废话和无意义装饰。
