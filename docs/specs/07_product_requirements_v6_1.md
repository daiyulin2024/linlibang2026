# 邻里帮 V6.1 产品需求文档

## 1. 产品定位

### 1.1 产品名称

邻里帮

### 1.2 新定位

邻里帮 V6.1 是一个面向社区老人的 **需求工单与邻里协同工具**。

它不再定位为“养老服务下单平台”，而是帮助老人用更低门槛表达需求，并让社区、邻里和认证服务者根据需求类型协同处理。

### 1.3 一句话说明

老人只要说清楚“我需要什么”，邻里帮就把它整理成工单，并判断优先由邻里、社区认证服务者还是社区工作人员处理。

### 1.4 调整原因

原方案涉及真实商家对接、志愿者调度、支付结算和服务履约，超出了课程项目可实现范围。V6.1 保留“社区、邻里、认证服务者共同帮助老人”的核心逻辑，但去掉在线支付和复杂商业闭环。

### 1.5 产品边界

本项目做：

- 老人需求提交
- 文字备注
- 语音输入与语音留言
- 图片上传
- 工单生成
- 风险等级判断
- 邻里/机构/社区三层分流建议
- 后台值班人员管理工单
- 认证服务者联系方式维护
- 费用记录，不做在线支付
- 邻里圈活动与互助内容审核

本项目不做：

- 在线支付
- 自动真实派单
- 服务完成承诺
- 真实资质审核
- 医疗判断
- 政策资格自动判断
- 真实隐私数据处理

## 2. 用户角色

### 2.1 老人

主要使用者。需要大按钮、少步骤、语音输入、照片辅助表达。

### 2.2 子女

辅助使用者。需要了解老人提交了什么需求、是否有风险、是否需要自己介入。

### 2.3 邻里互助者

社区内可提供低风险帮助的居民。适合处理顺路代拿、简单陪伴、结伴活动等事项。

### 2.4 社区认证服务者

经过社区备案或推荐的专业服务者，例如水电维修、家政保洁、管道疏通、理发、助浴护理等。

### 2.5 社区值班人员

后台管理者。负责查看工单、电话联系老人、转交认证服务者、审核邻里圈内容、记录费用。

## 3. 首页信息架构

首页保留 5 个一级入口，不再设置复杂二级分类。

### 3.1 生活需求

适用场景：

- 顺路代买
- 拿快递
- 买药
- 买菜
- 买饭
- 买日用品
- 其他生活小事

老人不需要先选择“买药/买菜/买饭”。点击“生活需求”后，直接通过文字、语音和照片描述具体需求。

### 3.2 上门事项

适用场景：

- 水电维修
- 管道疏通
- 家政保洁
- 理发
- 家电维修
- 助浴

重点是判断风险和推荐处理方。普通邻居不一定具备专业能力，因此上门事项通常优先推荐社区认证服务者或社区工作人员介入。

### 3.3 外出协助

适用场景：

- 去医院
- 去药店
- 去银行
- 去社区服务中心
- 散步
- 参加社区活动

系统根据是否涉及医疗、证件、钱款、行动不便等因素判断风险。

### 3.4 社区反馈

适用场景：

- 楼道灯坏
- 路面不平
- 电梯按钮看不清
- 公告字太小
- 坡道不好走
- 休息座椅不足
- 活动室不方便

该入口必须支持照片上传，用于让社区更清楚地理解问题。

### 3.5 人工帮助

适用场景：

- 老人不会描述
- 语音识别失败
- 事情比较紧急
- 需求涉及银行、医保、证件、医疗等高风险事项
- 老人主动选择“请社区联系我”

点击后直接生成后台待联系工单。

## 4. 统一需求提交页

所有一级入口点击后进入同一种页面结构。

### 4.1 页面字段

必须包含：

- 当前需求类型
- 文字备注框
- 语音输入按钮
- 语音留言保存
- 图片上传
- 紧急程度
- 是否需要社区电话联系
- 提交按钮

### 4.2 文字备注框

用于输入具体需求。

示例：

- 我想买和照片里一样的降压药，家里已经吃完了。
- 厨房水龙头一直漏水，地上很滑。
- 2号楼3层楼道灯坏了，晚上看不清。

### 4.3 语音功能

语音是核心功能，必须实现，不作为可选项。

V6.1 采用两层语音方案：

#### A. 语音转文字

老人点击麦克风，说出需求。系统将语音识别为文字，并自动填入备注框。

首版实现建议：

- 使用浏览器原生 Web Speech API。
- 支持普通话中文识别。
- 识别结果自动写入备注框。
- 识别失败时提示重新说一遍。

#### B. 语音留言

老人也可以留下一段原始语音，供后台人员直接听。

首版实现建议：

- 使用 `streamlit-mic-recorder` 录音。
- 保存音频字节或模拟保存录音状态。
- 工单中显示“有语音留言”。
- 后台工单看板可以看到该工单包含语音。

必须说明：

- 语音转文字用于快速生成备注。
- 原始语音用于防止识别错误，方便社区值班人员复核。

### 4.4 图片上传

所有需求类型都支持图片。

用途：

- 生活需求：药盒、购物清单、快递码、物品照片。
- 上门事项：漏水、电线、坏灯、损坏家电照片。
- 外出协助：预约单、医院单据、活动通知。
- 社区反馈：楼道、路面、电梯、公告栏等现场照片。

首版实现：

- 支持上传 1-3 张图片。
- 工单中显示“有图片”。
- 后台能预览图片或看到图片文件名。

### 4.5 紧急程度

选项：

- 不急
- 今天需要
- 比较紧急
- 很紧急，需要人工联系

如果选择“很紧急”，系统自动推荐社区工作人员优先处理。

## 5. 工单生成

老人提交后生成一张工单。

### 5.1 工单字段

- 工单编号
- 老人姓名
- 老人住址
- 需求类型
- 文字备注
- 语音转写文本
- 是否有原始语音
- 是否有图片
- 紧急程度
- 风险等级
- 推荐处理方
- 当前状态
- 创建时间
- 费用记录

### 5.2 风险等级

分为：

- 低风险
- 中风险
- 高风险

### 5.3 推荐处理方

分为：

- 邻里互助
- 社区认证服务者
- 社区工作人员

## 6. 三层分流规则

### 6.1 邻里互助优先

适合条件：

- 低风险
- 不涉及进入老人家中
- 不涉及银行、密码、验证码
- 不涉及医疗判断
- 可以顺路完成
- 老人同意进入邻里互助池

示例：

- 顺路买菜
- 代拿快递
- 陪散步
- 结伴参加活动

### 6.2 社区认证服务者优先

适合条件：

- 需要专业能力
- 需要上门处理
- 邻居通常无法完成
- 社区已有备案联系方式

示例：

- 水电维修
- 管道疏通
- 家电维修
- 家政保洁
- 理发

### 6.3 社区工作人员兜底

适合条件：

- 老人主动选择电话联系
- 需求描述不清楚
- 语音识别失败但有录音
- 紧急程度高
- 涉及银行、医保、证件
- 涉及助浴、就医等高风险事项
- 邻里长时间未响应
- 认证服务者长时间未响应

## 7. 响应超时机制

分流需要体现“实时处理”的思路。

### 7.1 邻里超时

如果工单进入邻里互助池后长时间无人响应，系统应转给社区认证服务者或社区工作人员。

演示版设置：

- 用“模拟邻里无人响应”按钮代替真实等待。
- 点击后工单状态从“邻里互助等待中”变为“邻里无人响应”。
- 系统自动推荐转给认证服务者或社区工作人员。

### 7.2 机构超时

如果认证服务者未响应，转社区工作人员后台代办。

演示版设置：

- 用“模拟机构未响应”按钮代替真实等待。
- 点击后状态变为“待社区电话确认”。

## 8. 费用记录

本项目不做支付，但保留后台费用记录。

### 8.1 费用原则

- 老人端不显示复杂支付流程。
- 工具不收款。
- 费用由后台工作人员记录。
- 可设定季度统一结算。

### 8.2 后台费用字段

- 工单编号
- 服务类型
- 服务者
- 预计费用
- 实际费用
- 是否已确认
- 结算周期
- 备注

老人端仅提示：

> 该事项可能产生费用，具体金额由社区工作人员或认证服务者线下确认。

## 9. 后台管理系统

后台是 V6.1 的关键组成部分。

### 9.1 工单看板

显示所有工单：

- 工单编号
- 老人姓名
- 类型
- 备注摘要
- 是否有语音
- 是否有图片
- 风险等级
- 推荐处理方
- 当前状态
- 创建时间

可操作：

- 标记已电话联系
- 转邻里互助
- 转认证服务者
- 模拟邻里无人响应
- 模拟机构未响应
- 标记完成
- 记录费用

### 9.2 认证服务者管理

每个类型保留一个社区认证服务者即可。

示例：

- 水电维修：安靖便民维修队
- 管道疏通：畅通管道
- 家政保洁：安心家政
- 理发服务：社区便民理发点
- 助浴护理：备案护理站

字段：

- 服务类型
- 机构名称
- 联系人
- 电话
- 是否可上门
- 是否需要社区陪同
- 备注

### 9.3 邻里圈审核

后台人员审核可公开内容。

可公开：

- 低风险互助需求
- 社区活动
- 共治反馈进度
- 好人好事

不可公开：

- 药品照片
- 完整门牌号
- 老人电话
- 医疗细节
- 银行事项
- 家庭隐私

### 9.4 活动发布审核

保留邻里圈活动资讯。

活动类型：

- 健康讲座
- 防诈骗课堂
- 手机使用教学
- 义诊
- 棋牌活动
- 志愿者巡检

活动状态：

- 待审核
- 已发布
- 已结束

## 10. 邻里圈

邻里圈保留，但重新定义为“社区活动 + 邻里互助 + 共治反馈”。

### 10.1 社区活动

展示后台审核通过的活动。

### 10.2 邻里互助

展示审核后的低风险互助需求。

示例：

- 2号楼有老人需要顺路拿快递。
- 周三下午有老人想结伴去活动室。
- 5号楼有老人想找棋友。

### 10.3 共治反馈

展示社区环境问题处理进度。

示例：

- 2号楼楼道照明问题：待处理。
- 公告栏字体太小：已记录。
- 活动室座椅不足：处理中。

## 11. 我的页面

老人端“我的”页面保留。

内容：

- 当前老人
- 切换老人
- 字体调节
- 我的工单
- 我的反馈
- 我的语音留言
- 紧急联系人

## 12. 演示主线

### 主线一：生活需求

1. 点击“生活需求”。
2. 老人通过语音说：“我想买这个降压药，家里吃完了。”
3. 上传药盒照片。
4. 选择“今天需要”。
5. 提交工单。
6. 系统判断：
   - 风险等级：中
   - 推荐处理：社区工作人员先电话确认，再进入邻里互助或子女协助
7. 后台看到该工单。
8. 后台标记“已电话联系”。

### 主线二：上门维修

1. 点击“上门事项”。
2. 输入：“厨房水龙头漏水，地上很滑。”
3. 上传照片。
4. 提交工单。
5. 系统判断：
   - 风险等级：中
   - 推荐处理：社区认证服务者
6. 后台转给“安靖便民维修队”。
7. 后台记录预计费用。

### 主线三：社区反馈

1. 点击“社区反馈”。
2. 输入：“2号楼3层楼道灯坏了，晚上很黑。”
3. 上传照片。
4. 选择允许公开处理进度。
5. 系统判断：
   - 风险等级：高
   - 推荐处理：社区工作人员
6. 后台审核后发布到邻里圈共治反馈。

## 13. 首版实现范围

必须实现：

- 首页 5 个入口
- 统一需求提交页
- 文字备注
- 语音转文字
- 原始语音留言
- 图片上传
- 工单生成
- 本地规则分流
- 后台工单看板
- 认证服务者列表
- 费用记录字段
- 邻里圈三板块
- 我的工单记录

暂不实现：

- DeepSeek API
- 在线支付
- 真实短信或电话
- 真实服务者登录
- 真实推送通知
- 政策匹配

## 14. DeepSeek 预留

DeepSeek 暂不接入首版。

后续可用于：

- 老人需求摘要
- 风险判断解释
- 工单公开版脱敏
- 给老人、子女、后台分别生成不同说明

但首版应优先完成语音、图片、工单和后台闭环。

## 15. 验收标准

首版完成后应满足：

- 老人不用二级分类也能提交需求。
- 每个一级入口都能生成工单。
- 工单能包含文字、语音状态、图片状态。
- 系统能给出风险等级和推荐处理方。
- 后台能看到工单并修改状态。
- 后台能记录费用但不收款。
- 邻里圈能显示审核后的活动、互助和反馈动态。
- 语音功能可实际使用，不只是静态按钮。
